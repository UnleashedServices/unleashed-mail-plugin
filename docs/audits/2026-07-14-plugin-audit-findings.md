## Dimension: `mcp-server`

**Assessment:** The review-synthesizer MCP server is a genuinely strong piece of engineering: the stdio JSON-RPC implementation is protocol-correct and battle-tested (159/159 tests pass, exercised as a real subprocess), the zero-dependency and determinism claims both verify, .mcp.json matches today's documented plugin mechanism exactly, and the server↔swift-reviewer contract is field-for-field accurate including the mcp__plugin_ tool namespacing. The gaps are at the edges rather than the core: one fail-open inconsistency (an explicitly empty changed_files silently converts a real blocker into a pre-existing APPROVE, contradicting the server's own documented threat model), documentation drift in the prose-fallback rules (README omits the three COREDEV-2332 ai-safety merge pairs the code implements), a dead API layer referencing a nonexistent reviewers.py, and the final-verdict math after the verify gate still being done in LLM prose despite decide_verdict existing in code. **Strengths:**

- MCP protocol implementation is unusually careful and is tested end-to-end as a real subprocess: notification-vs-request distinguished by id membership (explicit id:null still gets a reply), array/empty params rejected with -32602 instead of crashing to -32603, non-JSON and non-object lines tolerated without killing the loop, unknown-method errors suppressed for notifications, UTF-8 reconfigure on all three streams, and a readline() loop with a documented rationale for avoiding the file-iterator read-ahead deadlock. ## Dimension: `orchestration-architecture`

**Assessment:** The agent-utilization architecture is conceptually strong — a docs-valid nested fan-out (subagents can spawn subagents as of v2.1.172), a deterministic MCP synthesizer with an LLM verify gate, and a fail-closed status contract — but it is undermined by one fleet-wide mechanical defect and several silent-degradation paths. Most seriously, all 21 agents declare tool restrictions via `allowed-tools:`, which is not a supported agent frontmatter field (`tools`/`disallowedTools` are), so every least-privilege boundary, including the 'read-only' reviewers and the curated MCP whitelists, is silently ignored; relatedly, the SubagentStart/Stop capture hooks match bare agent names that plugin-scoped identifiers likely never hit, and 'ask the user' checkpoints live inside subagents that provably cannot use AskUserQuestion. Beyond correctness, the design leaves current capability unused (no `memory`, `skills` preload, `maxTurns`, or model tiering — 18 agents pinned to `opus`, now a downgrade from a Fable session), duplicates CLAUDE.md knowledge that subagents already auto-load, and hangs the entire mandated dev loop on two externally authenticated CLIs and fixed /tmp transcript paths with no degradation story.

**Strengths:**

- The core fan-out design is valid against today's platform: nested subagents are officially supported ('As of Claude Code v2.1.172, a subagent can spawn its own subagents', depth limit 5), so swift-reviewer legitimately dispatches five reviewers via the Agent tool, and its dual entry mode (spawn-yourself vs. accept pre-collected JSON arrays) degrades gracefully.
- Moving synthesis out of prose into a bundled deterministic MCP server (scope filter, dedup, ownership routing, provisional verdict in code) with an explicit LLM-owned verify gate on every blocker is ahead of common multi-agent practice, and the fallback-to-documented-rules path is spelled out.
- The Output Contract status enum (COMPLETE | BLOCKED | PARTIAL) read before the findings array correctly distinguishes 'couldn't run' from 'clean pass' — a genuine fail-closed semantic most review pipelines lack.
- hooks/hooks.json uses only valid event names (SubagentStart, StopFailure, PermissionDenied, PostToolUseFailure all verified against the official hooks docs) and the plugin ships its own static hook validator (validate-hooks.py) wired into CI — self-auditing infrastructure done right.
- scripts/lib/context.sh shows unusually careful file-handoff engineering for the code-review capture path: PII-safe slugs, per-checkout repo-hash namespacing, atomic writes, TTL'd bindings, fail-open observe-only semantics, and explicit avoidance of /tmp.
- Commands correctly use disable-model-invocation: true, keeping /implement and /pr-review descriptions out of always-loaded context while agent descriptions carry the auto-trigger role.
- AGENT_CONTRACTS.md as a single arbitration point for cross-agent boundaries — with a documented history of catching real inconsistencies (the @v-pinning grep-filter bug, the jira blocking-vs-parallel contradiction) — is a strong pattern other plugins rarely have.
